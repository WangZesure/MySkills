# Agent Framework Rule

## 1. 适用范围

本框架用于需求明确、工具链齐备、问题可拆分为控熵子任务的工程过程。

**人负责**：需求定义、框架搭建、最终验收。
**Agent 负责**：在框架内执行阶段化、迭代式的信息生成与实现。

## 2. 文件组织

```
[task]/
  rule.md              ← 本文件（全局规则，仅一份）
  backward/
    goal.md             ← 任务 I/O 规格（STABLE，较大修改需记 Change Log）
    testpoints.md       ← 测试点和调试输出   
  forward/
    reference.md        ← 设计范式和参考工作
    knowledge.md        ← 相关知识
    sections.md         ← 计划，分成哪些阶段，每阶段设计的输入输出规范是什么    
  runtime/
    plan.md             ← 当前迭代中的工作计划，因为什么要做什么，例如调试查看问题出现在流程中哪部分，迭代阻塞考虑重新审视设计并设计检查点，调研并填充框架等。
    memory.md           ← 迭代驱动型版本记录
    overview.md         ← 三层描述（接口契约 / 计算模型 / 实现设计）    
  workspace/            ← 业务产出目录
  templates/            ← 空模板（新任务初始化时从此复制）
```

## 3. 工作流
总体工作流是：从用户提供内容，如数据、工作环境等，和调研到的相关知识、设计范式、例子等信息前向计划；从用户需求，设计计划的子阶段接口反向设计检查点和调试输出。
然后实际进行设计，根据输出不断迭代，更新知识和设计框架，不断排除错误设计，提升对设计的整体认识；且丰富完善检查点，最终使得我们的实际设计满足所有检查点和最终需求。

### Stage 1: 调研

**动作**：
1. 读 `goal.md` 确认 I/O
2. 搜索现有方案、文献、参考实现
3. 整理到 `reference.md`（来源、相关性、关键发现、可提取元素）
4. 提取原子定义到 `knowledge.md`（定义、约束、设计影响、来源）

**产出**：reference.md + knowledge.md

### Stage 2: 配置环境

**动作**：
1. 在 `workspace/` 创建目录结构
2. 安装依赖、配置工具链、环境变量
3. 如有外部 skill，记录调用约定
尽可能使得agent可以直接操作和看到工作环境，而非人来运行和给出反馈。
**产出**：workspace/ 就绪

### Stage 3: 拆分子阶段

**判断标准**：无法在一次思维过程中完整设计,或有过多分支导致无法以数量可控的调试输出来测试(一般在多次迭代失败后发现)，则拆分。

**动作**：
1. 简单/强耦合任务 → 跳过本阶段
2. 复杂任务 → 填写 `/sections.md`，每个子阶段定义 I/O 契约和状态
3. 子阶段详细设计放在 `/overview-[X].md`（如 overview-A.md），供父级 overview.md Layer 3 引用

**产出**：sections.md（可选）

### Stage 4: 设计与测试

**动作**：
1. 编写 `/overview.md`（如需要），使用三层描述
2. 编写 `/testpoints.md`：
   - P0: 必须通过才能验收
   - P1: 应该通过，表示质量达标
   - P2: 最好通过，表示完备
3. 初始化进度文件：
   - 计划驱动 → `/plan.md`（步骤 + 依赖图，Verification 引用 testpoints 编号）
   - 迭代驱动 → `/memory.md` V1 条目

**产出**：overview.md（可选），testpoints.md，plan.md 或 memory.md

### Stage 5: 迭代

**每轮动作**：
1. **读状态**：从 `runtime` 了解当前进度
2. **执行**：完成下一步
3. **验证**：跑 testpoints（自动）或展示给人（人工）
4. **更新 /**：
   - 计划驱动 → 更新 plan.md 步骤状态
   - 迭代驱动 → 追加 memory.md 条目（时间戳、版本号、变更、状态、反馈、下一步）
5. **更新 forward/**：新理解追加到 knowledge.md
6. **按需更新 /**：设计理解变化时更新 overview.md

**终止条件**：全部 P0 testpoints 通过 + 人工确认。

**阻塞时**：回退到 Stage 3（拆分）或 Stage 1（调研）。即使前置信息和计划都做得很好，测试点设计的也较好，这部分仍然是重中之重，因为我们无法确定会有什么超出预期的状况。当你在一个地方迭代很多次，仍然不明确问题所在，只能进行猜测时候，就需要回退到反思(根据调研信息，当前设计，当前知识)，完善需求(设置更完善的检查点)这些过程，在实验过程中不断完善你的认识和需求。只有这两个单调增加，才能确保任务朝着成功的方向迭代。

### Stage 6: 交付

**动作**：
1. 整合产出到 `workspace/`
2. 对照 goal.md 逐项检查
3. 运行全部 testpoints
4. memory.md 追加最终版本条目

**关卡**：人工对照 goal.md 验收。

### 文件分类

| 分类 | 目录 | 规则 |
|------|------|------|
| STABLE | `goal.md,sections.md,backward/` | 经批准后稳定；修改需追加 Change Log |
| ACCUMULATED | `reference.md,knowledge.md` | 只追加，不删除；可标记过时 |
| MUTABLE | `runtime/,` | 每轮迭代按需更新 |

### 快速启动

根据任务类型选择入口：

| 场景 | 入口 |
|------|------|
| 已有代码基的增量任务 | Stage 4（ 只写 delta） |
| 纯调研 | Stage 1 → Stage 5（收集反馈） |
| 简单修复（单文件、明确改动） | 直接 Stage 5（跳过所有阶段） |
| 完整 greenfield | Stage 1 → 6 全部 |

### 按需创建

不预创建所有文件。初始化时按任务类型选择：

| 任务类型 | 需要的文件 |
|----------|-----------|
| 计划驱动 | goal.md + plan.md + testpoints.md |
| 迭代驱动 | goal.md + memory.md + testpoints.md |
| 混合型 | goal.md + plan.md + memory.md + testpoints.md |
| 需调研 | 追加 reference.md + knowledge.md |
| 复杂需拆分 | 追加 sections.md |
| 需完整设计文档 | 追加 overview.md |

## 4. 表达规范

### 3a. 三层描述标准（用于 overview.md）

**Layer 1 — 接口契约**：不变量、类型签名、前置/后置条件。不做自然语言描述。I/O 表引用 `/goal.md`，不重复。

**Layer 2 — 计算模型**：用最适合的形式化工具描述问题本质——状态机、数据流图、调用图、伪代码、类型系统、集合映射。选最自然的，不强制数学符号。

**Layer 3 — 实现设计**：模块划分、数据流、关键算法/规则。对非专家可读，要对应到代码库结构。

### 3b. 精度规则

- 判断须有可验证条件，禁止模糊自然语言
- 调研结论引用实际来源（URL、文件路径、论文编号）
- 术语首次出现时定义；区分 [事实] 与 [推测]
- 优先使用类型签名、状态转移等精确描述；自然语言仅作动机和上下文补充

### 3c. 文件格式

每种文件有固定条目格式（见 templates/ 中对应模板）。新增条目时遵循格式，不改变结构。

## 5. 文件变更规则

| 文件 | 规则 |
|------|------|
| `/goal.md` | STABLE；修改须在 Change Log 记录原因和日期 |
| `/sections.md` | 子阶段拆分变化时修改 |
| `forward/reference.md` | 只追加；可标记"已过时"，不删除 |
| `forward/knowledge.md` | 只追加，可标记"不全面"，不删除 |
| `/overview.md` | 设计变化时更新对应层 |
| `/plan.md` | 更新步骤状态；可新增步骤，不删除已完成步骤 |
| `/memory.md` | 只追加，不覆盖历史 |
| `/testpoints.md` | 可新增；不可降低已有标准的优先级 |

## 6. 任务类型分发

### 计划驱动型
**特征**：管线已知、步骤依赖明确。
**追踪**：`/plan.md` 为主体（步骤 + 依赖图）。Verification 引用 testpoints.md 编号。
**memory.md**：可选，记录关键决策偏离。

### 迭代驱动型
**特征**：设计开放性、需人工反馈、步骤非固定。
**追踪**：`/memory.md` 为主体，版本号递增。
**plan.md**：可选，记录阶段性目标。

### 混合型
同时使用 plan.md（依赖 DAG）+ memory.md（迭代变更记录）。

## 7. 会话交接协议

新会话按以下顺序读文件：

```
1. STATUS.md           → 秒级恢复上下文
2. rule.md             → 理解框架规则
3. /goal.md        → 理解任务目标
4. /           → plan.md（定位剩余步骤）或 memory.md（最近 3 条）
5. /overview.md  → 理解当前设计（如存在）
6. /testpoints.md → 理解验收标准
```

**上下文紧张时**：STATUS.md + goal.md + plan.md 当前步骤 / memory.md 最近 3 条 > rule.md（已缓存可略）> 其他按需读。

## 8. 子阶段管理

子阶段不再使用独立目录。所有子阶段信息集中在 `/sections.md`：

- 每个子阶段：名称、I/O 契约、状态、对应的  文件路径
- 详细设计：`/overview-[X].md`（如 overview-A.md）
- 父级 overview.md Layer 3 引用子级 

默认最多一层子阶段。更深递归需在 sections.md 标注并说明理由。
